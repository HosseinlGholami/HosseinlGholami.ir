System Overview

This project turns raw motion signals from vehicle‑mounted sensors into actionable safety insights. Using accelerometer and gyroscope data, the edge device segments streams into sliding windows and detects harsh events such as abrupt braking, aggressive acceleration, sharp turns, and lane changes. Detected events and metadata are published to the cloud in real time for storage, analytics, and trip grading.

System Architecture

The pipeline has three layers: on‑vehicle sensing and lightweight detection, a cloud ingestion/processing tier, and a scoring & dashboard layer. Vehicles publish over MQTT; RabbitMQ handles message brokering (MQTT plugin) and Redis provides fast in‑memory buffering. Long‑term data lands in PostgreSQL and the system is packaged in Docker for reproducible deployment and horizontal scaling.

Hardware Components

1 – In‑Vehicle IoT Device

  • Microcontroller / Phone: ESP32 or smartphone SoC
  • Sensors: 3‑axis accelerometer & 3‑axis gyroscope
  • Communication: MQTT over Wi‑Fi/BLE or cellular

2 – Data Uplink

  • Protocol: MQTT topics per vehicle/trip
  • Message Broker: RabbitMQ (MQTT plugin)
  • Temporary Storage: Redis streams/cache

Software Components

  • Algorithm: Sliding‑window analysis with FastDTW (template matching) + pruned decision tree (ID3) for event classification
  • Backend Workers: Python services for ingestion, labeling, aggregation, and scoring
  • Database: PostgreSQL for durable, queryable history
  • Orchestration: Docker & Docker Compose for simple deployment

Frontend Application

A web dashboard for fleet managers and insurers shows live event counts, trip timelines, and driver scorecards. Operators can drill down into a trip to review the sequence and severity of detected events.

System Monitoring

  • Real‑time MQTT throughput and broker queue depth
  • End‑to‑end latency (edge window → labeled event → persisted record)
  • Worker health, error rates, and device online status

Performance Highlights

  • Macro F1 on test set: ≈90% with a 186‑sample window and pruned decision tree (depth ≈12)
  • On‑device event detection: real‑time friendly via sliding windows + FastDTW comparison
  • Cloud ingestion stability: sustained processing under bursty uploads with Redis buffering

Conclusion

By combining efficient classical signal‑processing (FastDTW) with a lightweight decision tree and a lean, containerized backend, the system delivers accurate, real‑time driver behavior profiling on modest hardware. Next steps include exploring temporal deep networks for richer behaviors, broader calibration across vehicles/mounts, and extending scoring to insurance pricing and coaching feedback loops.
